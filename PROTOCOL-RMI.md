    <article class="markdown-body entry-content container-lg" itemprop="text"><h1><a id="user-content-comfocontrol-canrmi-protocol" class="anchor" aria-hidden="true" href="#comfocontrol-canrmi-protocol"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>ComfoControl CAN/RMI Protocol</h1>
<p>This document tries to outline the protocol as used by the newer "Q"-Models. Please note that some stuff might be version dependent, especially ranges. My findings are based on ~R1.6.2
Two basic assumptions:</p>
<ul>
<li>Your airflow unit is m³/h</li>
<li>Your temperature is measured in °C</li>
</ul>
<p>If your ventilation is set to something else you need to try it out</p>
<h1><a id="user-content-quick-overview-of-the-network" class="anchor" aria-hidden="true" href="#quick-overview-of-the-network"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Quick Overview of the network:</h1>
<p>Speed is 100kb/s, no CAN-FD but extended ID's are used (one exception: firmware uploading)
Each device has an unique node-id smaller than 0x3F or 64. If a device detects that another device is writing using "his" id, the device will change its id</p>
<p>Nodes send a periodic message to 0x10000000 + Node-ID with DLC 0 or 4</p>
<p>All PDO's (= sensors, regular data to be transferred, PUSH-Model) are sent with the following ID:
<code>PDO-ID &lt;&lt; 14 + 0x40 + Node-ID</code></p>
<p>Firmware-Updates are sent using 11-bit IDs or 1F800000</p>
<p>RMI-Commands are sent and received using extended-IDs:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="    1F000000

<h1><a id="user-content-encoding-an-rmi-commandmessage-into-can-messages" class="anchor" aria-hidden="true" href="#encoding-an-rmi-commandmessage-into-can-messages"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Encoding an RMI command/message into CAN-Messages</h1>
<p>There are two ways of messaging them:</p>
<ol>
<li>
<p>If the message is less or equal than 8 bytes, it will be sent unfragmented.
CAN-Data = RMI-Data
MultiMsg = 0</p>
<p>Example:
Request: <code>01 1D 01 10 0A</code> (Get (<code>0x01</code>) from Unit <code>1D 01</code> (<code>TEMPHUMCONTROL 01</code>) exact value (<code>10</code>) variable <code>0A</code> (<code>Target Temperature Warm</code>))
Response: <code>E6 00</code> little-endian encoded, <code>0x00E6 = 230 = 23.0°C</code></p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="    can1  1F011074   [5]  01 1D 01 10 0A
    can1  1F001D01   [2]  E6 00
"><pre><code>    can1  1F011074   [5]  01 1D 01 10 0A
    can1  1F001D01   [2]  E6 00
</code></pre></div>
</li>
<li>
<p>If the message is longer than 8 bytes, it will be fragmented into 7-byte blocks
Each block is prepended with the index (starting at 0) of the block. If the block is the last to come, 0x80 is added to the first byte.</p>
<p>Example:
CMI-Request: <code>80 03 01</code>, answer <code>00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00</code></p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="    can1  1F011074   [3]  80 03 01  # I send

    can1  1F005D01   [5]  84 00 00 00 00
</code></pre></div>
</li>
</ol>

<h1><a id="user-content-general-commands" class="anchor" aria-hidden="true" href="#general-commands"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>General commands</h1>
<p>There are three commands which always exist on a given Unit:</p>
<ul>
<li>
<p><code>0x01</code>: Read single property:</p>
<p>This command reads a single property identified by the Unit, SubUnit and Property Id.
Each writable property may have a definitionrange &amp; recommended step size which are also accesible</p>
<p>The syntax is: <code>01 UnitId SubUnitId GetType PropertyId</code>
The answer is depended on the type of the property and the GetType. If the value, range and step (everything) is requested, the answer is: Value, lower boundry of range, upper boundry of range, step size.</p>
<p>Example: Get maintainer password: <code>01 20 01 10 03</code>, answer: <code>34 32 31 30 00</code> (4210\0)<br>
Unit is <code>NODECONFIGURATION</code>, first SubUnit, PropertyId <code>03</code></p>
</li>
<li>
<p><code>0x02</code>: Read multiple property:</p>
<p>This command allows to read multiple propertys within one request, given that the caller knows the type of each.
As this command can be replaced by multiple calls to 0x01, we omit it here.</p>
</li>
<li>
<p><code>0x03</code>: Set a single property:</p>
<p>This commands sets one property to the given value. The value needs to be in the range as identified within 0x01, otherwise an error of 30 is returned. The step size, however, is not checked.</p>
<p>Syntax: <code>03 UnitId SubUnitId PropertyId Value</code></p>
<p>Example: <code>03 1D 01 04 00</code><br>
Unit is <code>TEMPHUMCONTROL</code>, first SubUnit, PropertyId: <code>04</code><br>
Sets the Property "Sensor ventilation: Temperature passive" to off</p>
</li>
</ul>
<p>All other commands are &gt;= 0x80 and dependent on the SubUnit.
One example for a custom, unit dependent command is 0x85 from Unit <code>SCHEDULE</code> (<code>15</code>). It disables a given Timer Entry for a timer, in the example it allows the bypass to be returning to its automatic position:<br>
<code>85 15 02 01</code></p>
<p>Please do not try to run command 0x80, 0x82 on NodeConfiguration (0x20), they will probably break your configuration, and even worse would be calling ANY &gt;= 0x80 command on 0x01. <br>
It can probably completely brick your ventilation. (it enters factory mode or tries to perform an update)</p>
